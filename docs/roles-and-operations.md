# Роли и операции в экосистеме GND

| KB @CerberRus00 - Nexus Invest Team 2026

## 1. Роли

### 1.1. Нода (GND_v1)

- **Назначение:** блокчейн-нода: хранение состояния, применение транзакций, консенсус, REST API.
- **Ответственность:**
  - Хранить и обновлять нативные балансы (GND, GANI) в таблице `native_balances` и в памяти (state).
  - Применять только разрешённые операции: перевод по подписанной транзакции, списание газа (GND), начисление при первом запуске (initial_mint).
  - Сохранять состояние в БД после применения каждого блока (нативные балансы и nonces), чтобы при перезагрузке данные восстанавливались из БД.
- **Ограничения:** изменение нативных балансов только через код ноды (core), без прямого доступа извне к записи в `native_balances`.

### 1.2. Админ-панель (GND_admin)

- **Назначение:** веб-интерфейс для администраторов (кошельки, токены, транзакции).
- **Ответственность:**
  - Запрос к ноде: список кошельков (GET /api/v1/admin/wallets с X-Admin-Token), балансы (GET /api/v1/wallet/:address/balance), отправка транзакций (например POST /api/v1/token/transfer с полем symbol для нативной монеты).
  - Не хранит и не получает private_key; отображает метаданные токенов (лого, адрес контракта) из ответа ноды.
- **Ограничения:** все операции с балансами выполняются через API ноды; админка только формирует и отправляет запросы.

### 1.3. Пользователь / Внешние системы

- **Назначение:** инициация переводов, просмотр балансов через API ноды или через админку/приложения.
- **Ответственность:** подписывать транзакции (через кошелёк/Signing Service); указывать symbol (GND или GANI) при переводе нативной монеты.
- **Ограничения:** не могут напрямую менять состояние; только через принятые нодой транзакции.

## 2. Операции над нативными монетами (GND, GANI)

| Операция            | Кто инициирует | Где выполняется | Защита / Условие |
|---------------------|----------------|-----------------|------------------|
| Перевод GND/GANI     | Пользователь/админка | Нода (ApplyTransaction, processTransfer) | Подпись, проверка баланса и nonce, symbol ∈ {GND, GANI} |
| Списание газа       | Нода (при вызове контракта) | Нода (ApplyExecutionResult) | Только в GND; при заданном fee_collector_address — зачисление на него |
| Начисление при первом запуске | Нода (InitFirstRun) | Нода + запись в native_balances | Один раз при инициализации генезиса |
| Чтение баланса      | Админка/клиент | Нода (GET /wallet/:address/balance) | Данные из state (native_balances + token_balances) |

## 3. Защита от кражи и сохранность

- **Изменение нативных балансов** возможно только через код ноды: `State.AddBalance` / `SubBalance` вызываются из `ApplyTransaction`, `ApplyExecutionResult` и `InitFirstRun`. Внешний API не предоставляет методов для прямого изменения `native_balances`.
- **Валидация транзакций:** проверка символа (только GND/GANI), суммы, достаточности баланса отправителя и (для газа) баланса GND; проверка nonce и подписи.
- **Сохранность при перезагрузке:** нативные балансы хранятся в PostgreSQL (`native_balances`); текущее состояние аккаунтов — в `accounts` (nonce, balance_gnd); снимки по блоку — в `account_states`; слоты storage контрактов — в `contract_storage`. Состояния хранятся в памяти и кэшируются. После применения каждого блока вызывается `State.SaveToDB(blockID)`: запись в `native_balances`, `accounts`, а при blockID > 0 — также в `account_states` и `contract_storage`. При старте ноды `LoadFromDB` загружает состояние из `native_balances`, `token_balances` и `accounts`.
- **Системный владелец контракта:** если в конфиге задан `gndself_address` и владелец контракта (`contracts.owner`) совпадает с ним, комиссия за газ при вызове контракта не взимается (см. config/native_contracts.json).
- **Сборщик комиссий:** если задан `fee_collector_address` в config/native_contracts.json, списанный с отправителя газ (GND) зачисляется на этот адрес в ApplyExecutionResult.
- **Рекомендации:** резервное копирование БД; мониторинг ошибок записи при `SaveToDB`; не логировать и не возвращать в API приватные ключи.
